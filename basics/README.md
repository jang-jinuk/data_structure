## 단일 연결 리스트
![](https://mudhub-bucket.s3.ap-northeast-2.amazonaws.com/gist/Screenshot+2025-11-06+at+11.46.56%E2%80%AFPM.png)

데이터를 순차적으로 저장하는 형태로, 각 노드에 다음 노드의 주소를 저장하는 방식으로 연결된 구조이다.

## 배열도 있는데 왜 단일 연결 리스트를??
단일 연결 리스트는 데이터를 **순차적으로 저장**한다는 의미에서 **배열과 딱히 다를 게 없지 않나?**라는 의문이 생겼다.
둘의 차이는 물리적으로 연결되어 있는지 논리적으로 연결되어 있는지에 대한 차이였다.
**배열**은 **물리적으로 연결**되어 있는 구조이고, **단일 연결 리스트는 논리적으로 연결된 구조**라고 할 수 있다.
배열은 **메모리상 실제로 이어져 있는 구조**로 되어 있는 반면에, 단일 연결 리스트는 물리적으로 떨어져 있더라도 위에서 말했듯이 **포인터 주소로 다음 위치를 저장**하여 논리적으로 연결된 구조인 것이다.

## 배열 VS 단일 연결 리스트
두 구조의 차이는 이해가 되었지만, 그렇다면 왜 배열 하나로도 충분할 거 같은데 연결 리스트를 사용하는 걸까?

### 배열
**장점**
- 메모리상 실제로 **연속된 공간에 저장되기** 때문에 **접근이 빠름**
- 인덱스로 **접근이 쉬움**

**단점**
- **크기가 고정**되어 있어, 처음 정해놓은 크기 이상을 넣지 못함
- 새 데이터를 삽입하면 뒤의 모든 요소가 한 칸씩 밀려 야하기 때문에 중간에 **삽입/삭제가 느림**

### 단일 연결 리스트
**장점**
- 필요할 때마다 동적으로 추가/삭제 가능하기 때문에 **크기 제한이 없음**
- 새 노드 삽입 시 포인터만 바꾸면 되기 때문에 **삽입/삭제가 빠름**

**단점**
- 인덱스로 접근 불가(**첫 노드부터 차례대로 조회해야 함**)
- 각 노드마다 포인터 **저장 공간이 추가로 필요함**(조금 메모리 낭비)

다음과 같이 두 구조에는 각각 장단점이 명확하기 때문에 특정 사용 환경과 목적에 따라 어떤 자료구조를 선택할지가 결정된다.
